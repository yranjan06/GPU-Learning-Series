What is latency hiding

*worked out from first principles*

Here's the strangest fact about a GPU. Its individual cores are not smart and they are not fast. Each one is a dumb little worker with a slow clock, far weaker than a single CPU core. And yet the GPU as a whole runs circles around the CPU on the right work.

It's like watching someone with no magic skills pull off the biggest magic trick in the room. The secret has a name, latency hiding, and once you see it you understand why a GPU is built the way it is.

The Delay Problem: Data takes time

Start with the problem everyone underestimates. Data takes time to travel.

Even on a small chip with expensive wide memory, when a core asks for a number from main memory, that number has to physically make its way across to the core, and that takes a while. Hundreds of clock cycles, sometimes. This delay is latency, and no amount of fancy memory fully erases it. The wire has a length and the trip takes time. So every core, GPU or CPU, constantly faces moments where it has asked for data and simply cannot proceed until it arrives.

Two Philosophies: Erasing vs. Embracing

Now the two designs answer this in completely opposite ways, and that's the whole story.

A CPU hates the wait. Its entire philosophy is, do not let the core sit idle, not for a microsecond. So it spends enormous resources making the wait disappear. Big caches keep data parked right next to the core so it rarely has to reach far. A branch predictor guesses what data will be needed next and fetches it early. Out-of-order execution lets it grab whatever work is ready and do that first instead of stalling. All of this is the CPU trying to erase the delay entirely so its one precious core never stops.

A GPU does the opposite. It doesn't try to erase the wait. It embraces it. The GPU shrugs and says, fine, the data will take a while, so I'll just go do something else in the meantime. It doesn't guess, it doesn't predict, it doesn't build big caches. It switches.

The Dhaba Analogy

Picture a dhaba where rotis take five minutes in the tandoor. That five minutes is the latency.

A CPU-style dhaba with one VIP table would have the customer sitting there fuming while the table stays empty for five minutes, so the CPU works frantically to pre-bake rotis so the wait never happens.

But a GPU-style dhaba has ten tables. Table one orders, and the roti will take five minutes. So the manager tells table one to wait, and instantly turns to table two whose order is ready and gets them going. Then table three. Then four. By the time he's worked through the other tables, table one's roti is done, and he swaps right back to them.

The kitchen, the part that actually cooks, never stops for a single second. It just keeps changing which table it's serving.

That's latency hiding exactly. The delay did not get smaller. Table one still waited its full five minutes. But because there were always other tables with ready work to switch to, the cook stayed busy the entire time.

On a GPU, when one warp asks for data and has to wait hundreds of cycles, the scheduler puts it to sleep and instantly wakes another warp that has work ready. By the time it cycles back, the first warp's data has arrived. This swapping is so fast it costs almost nothing, and the math units, the only part that matters, never go idle. The GPU hid the delay behind other work instead of erasing it.

Why not thousands of smart cores?

This also answers a question that should be nagging you. If a single smart CPU core is so good at avoiding delay, why doesn't a GPU just use thousands of those smart cores? Wouldn't that be the best of both worlds?

It would be a disaster, because of what makes a core smart in the first place. All that cleverness—the big caches, the guessing engines, the complex routing—takes up enormous space on the chip, and none of it does any actual math. It's all management.

If you filled a GPU with thousands of those bloated smart cores, the smart managers would eat nearly all the silicon and leave almost no room for the workers that do the multiplication. So the GPU does the ruthless thing. It throws the smart managers out, fills the freed space with thousands of cheap dumb math units, and recovers the lost cleverness through the switching trick instead. It doesn't need predictors when it can just hide the wait behind more warps.

And you can't escape this by just making one core faster either, because physics puts a ceiling on that. Push a single core's clock past a point and it generates so much heat and draws so much power that it would cook itself. So instead of betting everything on one genius core racing against a thermal wall, the GPU builds a factory of ten thousand basic workers. For work like AI and graphics, which is millions of simple identical multiplications, you don't need one brilliant mind making hard decisions. You need a huge crowd of ordinary hands lifting together. At that scale the crowd wins, every time.

The Catch: The Memory Pipe

There's one catch worth naming, because the whole trick depends on it.

Latency hiding only works if there's always fresh work to switch to. And fresh work means fresh data arriving. If the pipe feeding the chip is too narrow, the new data never shows up, every warp ends up waiting on the same starved supply, and there is nothing left to switch to.

When that happens, all the tables fall asleep at once and the magic collapses into idle cores starving. So the switching trick and a wide memory pipe are partners. One hides the wait, the other guarantees there's always something to wait with.

↳ The General Takeaway

So the takeaway is almost philosophical. The CPU fights delay by trying to never wait. The GPU accepts delay and stays busy through everything else it could be doing instead.

The CPU bets on one genius who refuses to be idle. The GPU bets on a crowd so large that someone is always ready to work.

And for the kind of massive, repetitive math that AI is made of...
the crowd that embraces the wait beats the genius that fears it.
